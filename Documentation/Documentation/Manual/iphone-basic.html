Unity iOS Basics

This section covers the most common and important questions that come up when starting to work with iOS.

Prerequisites

I've just received iPhone Developer approval from Apple, but I've never developed for iOS before. What do I do first?

A: Download the SDK, get up and running on the Apple developer site, and set up your team, devices, and provisioning. We've provided a basic list of steps to get you started.

Can Unity-built games run in the iPhone Simulator?

A: No. The iPhone simulator is not very useful for Unity because it does not simulate all inputs from iOS or properly emulate the performance you get on the iPhone/iPad. You should test out gameplay directly inside Unity using the iPhone/iPad as a remote control while it is running the Unity Remote application. Then, when you are ready to test performance and optimize the game, you publish to iOS devices.

Unity Features

How do I work with the touch screen and accelerometer?

A: In the scripting reference inside your Unity iOS installation, you will find some classes called iPhoneInput, iPhoneSettings, and a few others. These new iOS-specific classes provide the hooks into the device's functionality that you will need to build your games. Consult the iOS Input System page for more information.

Does iOS support Unity's Terrain Engine?

A: Terrains are not currently supported in Unity iOS because you need extremely low poly rendering for terrains on iOS. The terrain system is not designed for this purpose, and you can do much better by simply implementing extremely low poly terrains.

My existing particle systems seem to run very slowly on iOS. What should I do?

A: iOS has relatively low fillrate. If your particles cover a rather large portion of the screen with multiple layers, it will kill iOS performance even with the simplest shader. We suggest baking your particle effects into a series of textures off-line. Then, at run-time, you can use 1-2 particles to display them via animated textures. You can get fairly decent looking effects with a minimum amount of overdraw this way.

Can I make a game that uses heavy physics?

A: Physics are pretty expensive on iOS, and we recommend you use them as little as possible. Physics require a lot of floating point number crunching -- which is not ideal for iOS hardware. You should completely avoid MeshColliders if at all possible, but they can be used if they are really necessary. To improve performance, use a low fixed framerate using Edit->Time->Fixed Delta Time. A framerate of 10-30 is recommended. Enable rigidbody interpolation to achieve smooth motion while using low physics frame rates. In order to achieve completely fluid framerate without oscillations, it is best to pick fixed deltaTime value based on the average framerate your game is getting on iOS. Either 1:1 or half the frame rate is recommended. For example, if you get 30 fps, you should use 15 or 30 fps for fixed frame rate (0.033 or 0.066)

Can I access the gallery, music library or the native iPod player in Unity iOS?

A: Yes - if you implement it. Unity iPhone (with the Advanced license) supports the native plugin system, where you can add any feature you need -- including access to Gallery, Music library, iPod Player and any other feature that the iOS SDK exposes. Unity iOS does not provide an API for accessing the listed features through Unity scripts.

UnityGUI Considerations

What kind of performance impact will UnityGUI make on my games?

A: UnityGUI is fairly expensive when many controls are used. It is ideal to limit your use of UnityGUI to game menus or very minimal GUI Controls while your game is running. It is important to note that every object with a script containing an OnGUI() call will require additional processor time -- even if it is an empty OnGUI() block. It is best to disable any scripts that have an OnGUI() call if the GUI Controls are not being used. You can do this by marking the script as enabled = false.

Any other tips for using UnityGUI?

A: Try using GUILayout as little as possible. If you are not using GUILayout at all from one OnGUI() call, you can disable all GUILayout rendering using MonoBehaviour.useGUILayout = false; This doubles GUI rendering performance. Finally, use as few GUI elements while rendering 3D scenes as possible.

Device Graphical Capabilities

Can I use normal mapped objects in Unity iOS?

A: iOS is somewhere between the TNT2 and the original GeForce in fragment processing capabilities. It can add or multiply two textures together, but normal mapping is simply not supported.

What are some basic vertex count guidelines?

A: To achieve 30 fps on iOS, you should keep the per frame visible vertex count below 7000. This does not limit how many vertices you should have per scene. As long as you can ensure that you never have more than 7000 vertices visible at one time you, can guarantee a consistently high framerate. Take advantage of the new occlusion culling feature to ensure you stay in the 7000 vertices target.

General Performance

How I can improve application startup time on iOS?

Application startup time one device is determined by your application binary size and first scene size. Application binary size can be improved by minimizing script dependencies and enabling stripping (an Advanced license feature). More details on how to optimize binary size can be found here. User experience can sometimes be improved by adding a small first scene that simply shows a logo and then starts loading the real scene. Scene size can be reduced by optimizing assets.

When running on iOS my game framerate is bad. How I can improve it?

Your game framerate could be affected by many things. The cuase could be too complex geometry, use of OnGUI elements during gameplay, sub optimal scripts, too many .NET object allocations, etc. The first step towards optimization is to identify your performance bottlenecks. Internal profiler is a tool that can help you. The next step is to actually make optimizations (you can read more about it here).

Xcode

I am getting a lot of compiler errors in Xcode. What's going on?

A: Make sure that you have selected the "Device" mode and not "Simulator" from Project->Set Active SDK in the menubar.

Sometimes when I click "Build and Go", I see strange errors.

A: This is a bug in Xcode, and is non-destructive to your project. Simply click OK to bypass the message, and your game should function normally.
